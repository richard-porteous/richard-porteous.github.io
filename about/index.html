This site is mostly about coding, but not only! I try help you think like a programmer/developer/coder.

These tutorials do not help you reproduce what we have done, it does try to help you understand why we made certain choices, why we kept up-front design details to a minimum but found it critical to show the direction what we wanted.

If you think your code needs to be able to do everything then you need to read the principles of KISS (keep it simple stupid) yes harsh, but important to note we are here to learn not to pander to our pride, we can do that elsewhere.

If you think hacking a piece of code to get it working is acceptable for long term software then you should read about clean coding. And understand its ok not to have everything perfect but adapt as we go.

If you think Agile is the way to go, let me point out that Agile was made by managers with productivity in mind, not software developers with easy to maintain code in mind.

Oh and clean code has its place too. Most often performance takes front seat so we develop with clean code and use the perforamce enhancements when going into production.

I made this mistake in expensive software I might add. Most of the time we think our code needs to recover from all sorts of bad things, well remember that most User aimed code needs to tell them what they are doing or the file they are importing has errors or assumptions. Avoid your own assumptions too. So display a list of input errors etc. Some apps can do some atomatic recovery but not on critical things, OK.

I have made this site as a reference for you and me. I don’t want to learn things again if I have spent a few months away. That is why I make notes of the things that are important to me.

Explore some of these things and cut down the ‘unknowns’ that are so common in the usual tutorials where things are ass-u-me-d. Note. People tell me that assumptions make an ‘ass’ about ‘u’ and ‘me’ i.e. ‘ass’‘u’‘me’ = assume. Ok I spelled that out didn’t I.
