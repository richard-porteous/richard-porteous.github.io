Refactoring - Winforms using mvc

Date: 09.01.2022 - Author: Richard Porteous

This was more to follow along in another book when I started, but take a look it has proven useful to me. This post is dependant on the repository.

I like this choice as it can make switching out each form one at a time for WPF a little simpler with old apps, just remember WinForms is designed to work with events, and you may end up fighting the UI.

Refactoring has its place and is best when you are maintaing an application.

I am picking on winforms as it is

  1. quick for putting out prototypes
  2. still used more than it should be
  3. has very little structure of its own
  4. often becomes difficult to maintain very quickly

The code becomes difficult to manage at some point between prototype and production even on small apps. They can quickly become a mess of tangled code that no one wants to fix.

When you hear programmers say, “This app needs to be re-written”. Each case is different so often rewriting is not the best option.

Alternatives

If you dont plan on keeping winforms it is likely better that you simply treat your winforms app as a prototype and example code, and start fresh. HOWEVER sometimes you don’t have a choice.

MVC, MVVM etc

The MVC I show you here is the easiest to follow if you are new to this. There are other ways to implement this. I don’t show M (model) here, but you will get the point.

MVC (mvvm etc) allows you to split the code out of a (WinForms) app with minimal impact. i.e. break some dependencies. You are able to do one small part at a time with refactoring so it can become part fo your maintenance.

Deal with BUGS especially those firing events

If using a datagridview you will likely find some code is trying to step through the grid instead of the data. A rookie mistake, which causes a lot of events to fire. These events will cause threading bugs. This will need to be fixed first as almost any change you make will break the code. A quick solution is using the copyto if its available.

WHY Refactor?

Mostly dependacies. Code becomes hard to fix, and one change affects code in another part - OR - you simply want to add features to the app but are unable to do so.

EXAMPLE PROJECT

This small project interval refactor project takes a WinForms C# app called Interval and adds MVC design to it. Martin Fowler (2019) describes this process with a Java app.

The app has a very simple UI. It doesn’t use a database, so you won’t see much in the model, a service, or dependency injection.

  1. The project starts with the 2nd commit with IntervalView which is the code we will modify.
  2. The next commit adds the IntervalModel and IntervalController etc.

The objective here is move code/logic/data out of the ui unless its ui specific.

  1. WinForms is the UI
  2. controller or view-model class
  3. model class.

Simply put …

UI Class. (WinForms) View or display. We let the forms load as normal. The code deals with UI-related things, and it makes calls to the controller when something on the UI is changed by the user.

Controller Class. business logic. Anything we would need that wouldn’t be affected by switching databases or UI.

Model Class Deals with getting and storing the data.

This project simply shows you how to add the controller (and a view interface) and the model. It also goes briefly into how the screen will be updated. See the project readme for more explanation.

For existing projects, create the controller and model for each WinForm and an Interface for the controller. When a WinForm is a popup or a seperate edit screen it could use the same controller and model - i.e. “Customers” and “CustomerAdd” are two different WinForms with much in common. General

When you make these changes take small steps. Separate the code in the project one variable at a time. Move the code into a function that needs to be moved to the controller or model. Change Globals into parameters etc. Then move the function. Test every step.

Read a book on refactoring if needed.

Your code should be easy to read and self-explanatory. Comments should not say the same as WHAT the code does, but rather explain WHY. Function names should say what the code is intended for and the function should only do that.

Avoid being clever in your code. Either you will come back in a year and wonder why you did it that way or some other programmer will come along and criticize you for being cryptic and simply rewrite that code.
