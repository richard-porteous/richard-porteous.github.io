## THIS ~~BOOK~~ tutorial

This tutorial teaches as we go, but there is so much you can learn at this point. In the chapters is what you need for the lessons. The back of the book is dedicated to *STUFF* that is important but can just get in the way. Please don't ignore those *chapters*.

I included notes in the back on basic tools like GIT, and what parts of the manual to read. This isn't to replace what is available online. Its will either make you aware of it, or add clarity.

### *Who this book is for*
if you answer yes to any of the three questions below this book may be for you.

+ Do you want to start making games and have a computer?
+ You have never made a game or you seem to get stuck every-time you try.
+ Are you unsure where to begin?

### *What you need*

Yourself, a computer, this book. Ok you don't need this book but it will help.

Defold requirements are quite low, especially for 2D. The games in this book are designed to be developed on low end computers. Development always uses more power than the same game compiled, so if buying a pc make sure you don't buy too close to the minimum.


### *We will learn*

When we are done you should be comfortable enough to continue and write some games of your own. 

+ The core-concepts. This is essential.
+ A few basic game design principles during each project.
+ Just enough Lua to prevent confusion
+ Some basics and a few useful tips for the defold editor. 
+ Construct a game*


Let me clarify, for time and interest, most projects here are started but not completed. We do make 1 complete game (though not polished or published). There are a lot of tutorials showing you how to make a playable game, we just fill the gaps they don't.

We use the current defold editor.

*NOTE:* for auto-complete you need to use *Ctrl+Space*. thats Control and the spacebar.
 
*ADVICE:*  Please, there is no right or wrong way to make a game, though some ways are easier to write and others easier to maintain, this is about learning. The general consensus is that code must be as easy to read as a book. Clarity is king. Even if certain practices were considered good a few years back, they may be failing to live up to expectations today. Hindsight is often the best teacher i.e. learn by making mistakes. If you do plan in making this a career, name the bits and pieces clearly, write your code for the next person to be able to read. Avoid trying to look too clever, or it will come back and bite you. Don't get caught in the current hype, but do follow best practice.


This book also adds a little extra from my experience in general programming. Not everything will be perfect as we go, and I will show you how to do some basic debugging and how to fix the bugs and some re-factoring which means to move code around. 

You'll find I repeat myself in this book usually to add another layer of understanding, but sometimes just to remind you.
